# Mobile Web Rebuild Roadmap

Updated: 2026-04-23

## Why this needs a rebuild

The current mobile experience is not really a maintained mobile app. It is a generated preview shell built from the desktop page and then patched with:

- `scripts/build-pages.mjs` string-rewriting the main HTML into `/mobile/index.html`
- `web/public/mobile/mobile.css` overriding desktop layout at large scale
- `web/public/mobile/mobile-aid-bar.js` injecting replacement controls into the DOM after load
- `web/public/mobile/sw.js` and `manifest.webmanifest` treating the result like a mobile/PWA surface

That approach is fragile. It makes mobile depend on desktop markup structure, CSS selectors, and button IDs staying stable forever. A rebuild should treat mobile as a first-class browser experience, not a patched clone.

## Product goal

Build a real mobile web client for phone browsers that shares domain logic and data contracts with desktop, but owns its own layout, interaction model, and rendering flow.

## Non-goals

- Do not build an Android-native app.
- Do not preserve every desktop interaction pattern if they are hostile on mobile.
- Do not keep the current generated `/mobile` shell architecture.

## Core problems to fix

- layout is adapted, not designed
- DOM patching depends on brittle IDs and markup order
- desktop CSS and mobile CSS fight each other
- action density is too high for phone viewports
- print-oriented controls are awkwardly ported
- install/PWA support is bolted on top of a shell that is not mobile-first

## Target architecture

### Shared layers

Keep shared:

- seed contract
- game data loaders
- parsing / formatting helpers where actually reusable
- play-state and linked-play pure logic
- cloud/auth service wrappers

### Mobile-specific layers

Build separately:

- `web/public/mobile/index.html`
- `web/public/mobile/app.js`
- `web/public/mobile/styles/`
- `web/public/mobile/components/`
- `web/public/mobile/features/`

The mobile client should import shared logic modules, not shared desktop DOM/rendering modules.

## UX direction

### Mobile priorities

- fast seed lookup
- clean roster overview
- quick JSON/share actions
- touch-friendly play mode
- clear linked-match status
- compact but readable player aid

### UI rules

- one primary task per screen section
- bottom-docked or thumb-reachable actions where appropriate
- no dependence on hover
- no tiny tab labels that require horizontal precision
- modals only when clearly justified
- use progressive disclosure instead of dumping the whole desktop UI onto a phone

## Proposed information architecture

### Top-level nav

- Lookup
- Roster
- Play
- Aid
- More

### Lookup

- seed input
- recent seeds
- favorites
- import JSON

### Roster

- faction + points summary
- unit groups as stacked cards
- expandable details
- quick copy/share/export

### Play

- score/resources
- round + phase
- large tracker controls
- unit tracker list optimized for touch
- linked status banner

### Aid

- searchable/filterable tactical cards
- collapsible faction/unit aid sections
- compact print/export options separated from the normal reading flow

## Work phases

## Phase 1: Mobile audit

Deliverables:

- current mobile pain list with screenshots
- map of which logic can be shared and which UI must be replaced
- list of desktop dependencies the current mobile shell relies on

Acceptance criteria:

- clear go/no-go on each shared module
- explicit list of mobile-only rebuild areas

## Phase 2: Shared logic boundary cleanup

Before rebuilding the mobile UI, move reusable logic out of desktop-specific code.

Examples:

- seed loading services
- cloud auth/session services
- play-state selectors
- linked-play sync adapters
- serialization helpers

Acceptance criteria:

- mobile app can import shared logic without importing desktop render code

## Phase 3: New mobile shell

Build a dedicated `/mobile/index.html` and `/mobile/app.js`.

Acceptance criteria:

- no HTML string rewriting from desktop build output
- no mobile DOM surgery script needed just to make controls usable

## Phase 4: Screen-by-screen rebuild

Suggested order:

1. Lookup
2. Roster summary/details
3. JSON/share actions
4. Play mode
5. Player aid
6. PWA/install polish

Acceptance criteria:

- each screen works independently on phone-sized viewports
- each screen has a clear primary action path

## Phase 5: Cutover and delete

- remove generated mobile shell logic from `scripts/build-pages.mjs`
- remove obsolete mobile patch CSS
- remove DOM surgery helpers that only existed to patch desktop markup on mobile

Acceptance criteria:

- mobile route is first-class, not generated by mutation
- old mobile patch system is gone

## Verification checklist

- iPhone Safari-sized viewport
- Android Chrome-sized viewport
- portrait and landscape smoke check
- install prompt / manifest / service worker still valid
- seed lookup works
- JSON view works
- linked play basic flow works
- touch targets are comfortable and consistent

## Definition of done

- mobile no longer depends on desktop HTML shape
- `mobile-aid-bar.js` is deleted or reduced to a narrowly justified enhancement
- `mobile.css` becomes mobile-client styling, not a giant override pack
- `/mobile` can evolve without breaking desktop

## Prompts

### Prompt 1: Mobile product/UX plan

```text
You are planning a full mobile web rebuild for the StarcraftTMG app.

Context:
- The current `/mobile` experience is generated from the desktop page and then patched with mobile CSS and DOM manipulation.
- It is outdated, brittle, and should be treated as a rebuild, not a polish pass.
- This is for phone browsers, not Android native.
- Shared domain logic should remain shared, but mobile should own its layout and interaction model.

Tasks:
1. Audit the current mobile implementation and explain why it is fragile.
2. Propose a mobile-first information architecture.
3. Identify which modules should be shared with desktop and which should be rebuilt specifically for mobile.
4. Produce a phased rebuild plan with clear deliverables and acceptance criteria.
5. Call out desktop assumptions that must not leak into the new mobile app.

Output format:
- current-state problems
- target UX
- technical architecture
- phased plan
- risks
```

### Prompt 2: Mobile implementation prompt

```text
Rebuild the StarcraftTMG mobile web client as a first-class `/mobile` experience.

Requirements:
- create a dedicated mobile entrypoint and layout
- reuse shared logic only where the abstraction is clean
- do not depend on desktop HTML string rewriting
- do not patch desktop markup with mobile DOM surgery as the primary approach
- optimize for phone browsers and touch interactions
- preserve the existing seed/data contract

Implementation order:
1. establish shared service boundaries
2. create the dedicated mobile shell
3. implement lookup flow
4. implement roster screen
5. implement play screen
6. implement aid screen
7. restore PWA/install behavior cleanly

Verification:
- test on mobile browser emulation after each milestone
- keep desktop unaffected
```

### Prompt 3: Mobile review prompt

```text
Review the StarcraftTMG mobile web implementation as a mobile-first product, not as a desktop app squeezed into a narrow viewport.

Focus on:
- touch ergonomics
- information density
- brittle dependencies on desktop DOM/CSS
- navigation clarity
- install/PWA reliability
- performance on phone-sized viewports

Deliver:
- findings ordered by severity

- concrete fixes
```
